1 System Overview

Two-project Dagster code location system. The dbt canonical pipeline builds a canonical dimensional model on top of the source trading system (SQL Server), materialised as Iceberg tables in Dremio. The Dagster hub is a central K8s orchestration control plane that federates the canonical pipeline and other data projects via gRPC, providing a single UI, PostgreSQL-backed run history, and K8s job execution across all locations.

Data Source — SQL Server

  • Source system schema
  • 105+ registered source tables
  • Trading: trades, deals, parcels, voyages
  • Financial: invoices, claims, swaps, options
  • Reference: currencies, countries, desks, grades
  • Read via Dremio SQL Server connector

dbt Canonical Pipeline — Code Location

  • dbt-core 1.10.20 + dbt-dremio 1.10.0
  • Dagster 1.12.21 — gRPC code server, port 4004
  • 431 dbt models across 5 layers
  • 8+ Dagster jobs, 5 schedules, 1 failure sensor
  • Canonical SQL views (27 dims, 51 facts)
  • Test harness + email reporting

Dagster Hub — Orchestration Control Plane

  • Dagster 1.12.21 — K8s webserver, port 3000
  • Federates 3–6 code locations via gRPC
  • PostgreSQL 15 — run/event/schedule storage
  • K8sRunLauncher — spawns K8s jobs per run
  • QueuedRunCoordinator — max 10 concurrent
  • DagsterDaemonScheduler

Two-Layer Design
canonical_dm/v4/ — Raw Dremio SQL views (source of truth for business logic). Direct table refs like SourceDB.schema.tblTradesHeader. Not dbt Jinja.

models/ — dbt execution layer wrapping canonical logic. Uses {{ ref() }} and {{ source() }} for Dagster asset graph construction and lineage.
Execution Order & Configuration
Sequential: tables → vdata → dimsmodel → factsmodel → viewsmodel

dbt config: 8 threads, profiles.yml for dev/qa/preprod/prod targets. Dremio port 9047 (SSL). Object storage: S3. Schema: materialize (tables), canonical.v4.dims / canonical.v4.facts (views).

5 Key Architecture Patterns
  • 🔌
    gRPC Code Location FederationThe Dagster hub loads each project as an external gRPC server (workspace-{env}.yaml). The canonical pipeline code location runs dagster code-server start -m dagster_definitions.definitions -h 0.0.0.0 -p 4004 inside its own K8s pod. The hub connects at runtime — no code bundling required.
  • 🏗️
    Two-Layer dbt Designcanonical_dm/v4/ holds the source-of-truth Dremio SQL (raw, no Jinja). models/ is the dbt execution wrapper using {{ ref() }} / {{ source() }}. Business logic changes happen in canonical SQL; dbt layer drives Dagster asset graph and lineage only.
  • 🏷️
    Tag-Driven Job SelectionEvery dbt model is tagged (source_tables, vdata, dimsmodel, factsmodel, viewsmodel, qv). Dagster jobs use AssetSelection.tag() — no manual asset enumeration. Adding a new model with the right tag automatically includes it in the correct job.
  • 🧊
    Iceberg-Backed Source Tables97 source table models are materialised as Apache Iceberg tables on S3. This gives ACID transactions, time-travel, schema evolution, and efficient incremental loads. Downstream vdata, dim, and fact models read from Iceberg via Dremio views.
  • 📊
    Audit Macro System9 dbt macros provide audit logging: stage_insert_audit_log, stage_audit_loader_to_audit_main, get_last_load_time, get_source_rowcount / get_target_rowcount. Source vs. target row counts compared and logged per run for data quality tracking.
  • 🧪
    External Test Harnessscripts/test_harness.py validates Iceberg materialisations against SQL Server source: row count comparison (threshold %), column metadata check, SELECT accessibility. Outputs HTML/JSON report, optionally emailed. Runs as a scheduled Dagster job (10:00 & 18:00 UTC).
