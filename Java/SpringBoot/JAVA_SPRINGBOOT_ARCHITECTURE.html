Laboratory Data Upload & Integration Service

Message-driven file-processing microservice that ingests laboratory test data (Excel/CSV), validates and enriches records via SOAP/REST services, and updates the core inspection & measurement system — Java 11, Spring Boot 2.7, Spring Integration.

1 System Overview

A production-grade file-driven integration service that automates laboratory test data upload into a core trading platform's inspection & measurement system. An inbound file poller watches a local directory, routes files by type (result data vs. certificates), parses multi-format inputs (Excel, CSV), validates against business rules, enriches via reference data lookups, and persists to the inspection system via SOAP web services. Certificates are archived to a document management system.

Inbound Layer

  • Spring Integration file poller (60s interval)
  • Filename-pattern-based routing
  • Factory-pattern file parsing (Excel/CSV)
  • Hibernate Validator pre-validation
  • Multi-format: XLSX, XLS, CSV, PDF stub

Processing Layer

  • Spring Integration message channels
  • Dedicated handlers per file type
  • Reference data enrichment (labs, UOMs, grades)
  • Assignment/lot matching via Logistics SOAP
  • Single-threaded executor (sequential safety)

Outbound Layer

  • Inspection SOAP service (create/update records)
  • Logistics SOAP (assignment queries)
  • Document management REST (certificate upload)
  • Reference Data REST (lookups, caching)
  • SMTP email notifications (HTML reports)
File-driven event architecture
No REST API endpoint triggers processing — a Spring Integration poller watches a shared directory, enabling seamless integration with upstream file delivery systems (SFTP, email extractors).
Per-record error isolation
Each row in an input file is processed independently. Failures on individual records do not reject the entire file — valid records proceed, failures are reported via email.
Multi-service enrichment pipeline
Raw file data is enriched via 5+ external service calls (reference data, logistics, trade) before inspection system update — ensuring data integrity at source.
Sequential processing for data consistency
A single-threaded executor ensures record updates to the same assignment are never processed concurrently, preventing race conditions in the inspection system.

Why file-based, not API-driven? Laboratory systems and upstream extractors deliver data as files (via SFTP/email). The file poller pattern decouples the service from delivery mechanism specifics, and filename conventions encode routing metadata (lab name, assignment ID, file type) without requiring API contracts.
3 Message Handlers

Spring Integration routes files to dedicated handlers based on filename prefix. Each handler is a ServiceActivator bound to a named channel. A single-threaded ResultExecutor queues processing tasks sequentially to prevent concurrent updates.

File Handler (Result)
Channel: resultChannel
Parses RESULT/DATES files via factory-pattern readers (Excel, CSV). Validates each row with Hibernate Validator, enriches via reference data/Logistics lookups, then queues a ResultProcessor task for inspection system update.
Document Handler (Cert)
Channel: certChannel
Handles certificate PDF files. Authenticates with the document management system, creates a folder structure by assignment, uploads the file with metadata JSON, and sends a status email.
Unprocessed File Handler
Channel: unprocessedChannel
Recovery handler that scans the unprocessed directory for files that failed initial routing. Re-queues them for processing — prevents silent data loss from transient startup failures.

5 Key Architecture Patterns
  • 📩
    Message-Driven File Processing (EIP)Spring Integration poller + channel + handler pattern decouples file arrival from processing logic. Routing by filename prefix, not content inspection — fast and deterministic.
  • 🔄
    Factory Pattern File ParsingFileParserFactory returns the correct reader (Excel, CSV, PDF) by file extension. Adding new formats requires only a new IFileParser implementation — no routing changes.
  • 🛡
    Per-Record Error IsolationEach entity carries its own error state. Processing continues for valid records even when others fail — no batch rejection, graceful degradation with full error reporting via email.
  • ⚡
    Sequential Executor for Data SafetyResultExecutor wraps a single-threaded ExecutorService. Tasks queued sequentially prevent concurrent inspection SOAP updates to the same assignment — eliminates race conditions.
  • ⇄
    Multi-Layer Validation PipelineThree validation stages: Hibernate annotations (format/nullability), business rules (SRD/assignment existence), processing rules (lot matching, UOM compatibility) — each stage filters before the next.
  • 💾
    Resilient Retry with BackoffAll external service calls (SOAP + REST) implement retry logic — 5 attempts, 60s delay on HTTP 503 and socket timeouts. Exhaustion throws RuntimeException for error reporting.
  • 🌎
    Maven Profile-Based Multi-EnvironmentEnvironment-specific properties copied from profile folders (dev/qa/uat/prod) at build time. Jasypt encrypts credentials at rest — master password provided at runtime via JVM arg.
  • 🔒
    Health-Check Gate Before ProcessingServerHealthCheckService pings all downstream endpoints (Inspection, Logistics, Reference Data) before processing begins — prevents wasted parsing and enrichment when services are unavailable.

8 Observability
Correlation Logging
Each file generates a unique ctxId hash propagated through all Spring Integration channels and handlers. Log entries traceable end-to-end per file.
Health Check Gate
ServerHealthCheckService pings Inspection, Logistics, and Reference Data endpoints before processing. Prevents wasted work when downstream services are unavailable.
Email Notifications
HTML status reports sent per file: line-by-line results table, error details, and CSV export. Separate emails for validation failures and certificate upload status.
File Archival
Every processed file moved to a timestamped archive directory. Provides audit trail and enables manual reprocessing if needed.
Structured Logging
Logback-based logging with per-environment configuration. On-host logs written to application log file; container deployments forward to centralized log aggregation.
Error Accumulation
Per-entity error fields concatenate all failure reasons across validation, enrichment, and processing stages — full diagnostic context in a single email row.
