Enterprise Integration Middleware

Event-driven middleware bridging a core trading platform with downstream operational services, built with Java 17, Spring Boot 3, Spring Integration, and Spring Batch.

1 System Overview

A production-grade async middleware service acting as the integration layer between a core trading system and an operations platform. It receives inbound REST requests, persists them immediately to Oracle, and processes them asynchronously via Spring Batch, routing each operation through a dedicated Spring Integration message channel to its handler, which delegates to a downstream REST proxy. Results are stored in Oracle and delivered back via callback or polling.

Inbound Layer

  • Spring MVC REST Controller
  • Bean validation (@Valid)
  • MapStruct DTO mapping
  • Persist to Oracle (PENDING)
  • Return HTTP 200 immediately

Integration Layer

  • Spring Integration DirectChannel
  • DynamicRecipientListRouter
  • One MessageHandler per operation
  • ServiceHelper (cross-cutting logic)
  • Spring Retry on all proxy calls

Outbound Layer

  • Generated OpenAPI client stubs
  • Hand-written proxy wrappers
  • Apache HttpClient5 (SSL-aware)
  • Reference data invoker (cached)
  • EhCache 3 (cache-aside pattern)
Decoupling intake from processing
Callers needed instant acknowledgement regardless of downstream latency. Solved with an outbox pattern, persist first, process asynchronously via Spring Batch.
Operation-level routing without branching code
Routing dozens of distinct operations cleanly. Solved with one DirectChannel and one MessageHandler per operation, wired via configuration, no giant switch statements.
Resilience against transient downstream failures
Downstream services occasionally time out or return 5xx. Solved with Spring Retry at the proxy layer with configurable attempt counts and back-off delays per service type.
Zero data loss on JVM crash
Mid-processing crashes left records in IN_PROGRESS state. Solved with a startup CommandLineRunner that resets stale records to FAILED for re-inspection or retry.

(one per operation) | OperationHandler | |- EhCache lookup (reference data) |- Call downstream REST Proxy (Spring Retry) |- Update Oracle [status = SUCCESS / FAILED] +- POST result to callback endpoint (optional)
Why not a message broker? In-process Spring Integration channels provide transactional consistency with the Oracle state store, zero broker operational overhead, and deterministic per-operation routing, all configured via properties, not code.
3 Spring Batch Jobs

Three batch jobs handle all asynchronous processing. Jobs are not auto-started, triggered programmatically via JobLauncher or by @Scheduled tasks, giving full control over execution timing and concurrency.

Integration Job
Scheduled / Programmatic
Main processing job. Reads PENDING request records from Oracle (up to 1500 per cycle), groups them by priority, and submits each group to the Spring Integration channel layer via a parallel ExecutorService thread pool.
Reference Data Sync Job
Scheduled
Synchronises reference data into a local Oracle cache table. Drastically reduces per-request lookup latency during batch cycles. Cached entries are served from EhCache; the sync job keeps them fresh.
Purge Job
Scheduled
Deletes completed and failed request records older than a configurable retention period (default 120 days). Bounds Oracle table growth without manual DBA intervention.
4 Database Design & Request State Machine

Oracle is the system of record for every inbound request. Each API call gets a row in the request table immediately on receipt, before any async processing begins. A state machine governs the lifecycle of every record.

Inbound
API Call		 → PENDING Batch→ IN_PROGRESS proxy→ SUCCESS or FAILED purge→ DELETED

On JVM restart: any IN_PROGRESS record is immediately reset to FAILED by a startup CommandLineRunner. Purge job removes SUCCESS / FAILED records older than the retention window.

5 Key Architecture Patterns
  • ⚡
    Async Request-Reply (Outbox style)Caller gets an immediate HTTP 200. Oracle acts as the durable outbox. Spring Batch picks up and processes records asynchronously, fully decoupling intake from processing latency.
  • ⇄
    Property-Driven Message RoutingA DynamicRecipientListRouter resolves the target DirectChannel at runtime from an externalized property. Routing can be changed without any code change or redeployment.
  • 🔄
    Priority-Based Parallel ProcessingSpring Batch groups pending records by a priority field. An ExecutorService processes each priority group concurrently, blocking before advancing to the next group, ensuring high-priority operations are never starved.
  • 🛡
    Startup Recovery (Fail-Safe)On every JVM start, a CommandLineRunner marks any IN_PROGRESS records as FAILED, preventing silent data loss after unexpected restarts.
  • 💾
    Cache-Aside (EhCache 3)Reference data is cached in EhCache backed by Spring Cache abstraction, eliminating redundant round-trips during high-frequency batch cycles.
  • 🔒
    Proxy Abstraction LayerGenerated OpenAPI client stubs are wrapped in hand-written proxy classes that add SSL, retry, error mapping, and logging, insulating business logic from generated code churn.
  • 🌎
    Single-Artifact Multi-EnvironmentOne JAR deployed across all environments. Spring Profiles plus Jasypt-encrypted credentials externalize all config, no rebuilds, no environment-specific artifacts.

8 Observability
Distributed Tracing
OpenTelemetry Java Agent auto-instruments HTTP, Spring Integration channels, and Spring Batch jobs. Trace context propagated end-to-end.
Metrics
Micrometer + Prometheus. JVM, HTTP, batch, and integration metrics exposed at /actuator/prometheus.
Logging
Logback externalized outside JAR, runtime log-level changes without rebuild. Forwarded to Splunk from host or container stdout.
Health Check
Spring Boot Actuator at /actuator/health. Custom smoke-test endpoint verifies all downstream services are reachable.
API Docs
Springdoc OpenAPI 2.6, Swagger UI auto-generated from controller annotations at /swagger-ui.html.
Startup Recovery
CommandLineRunner marks any IN_PROGRESS records as FAILED on restart. No request silently abandoned after a crash.
